Vertikální metriky

Tutoriál
by Rainer Erich Scheichelbauer
cz fr zh

1 September 2025 Publikováno 10 July 2012

Vertical metrics determine the first účaří in a text, the distance between lines of text, and the padding to the following object below the last účaří.

For historical reasons, there are no less than three sets of values that deal with your vertical metrics inside a font file. They’re known as hhea, typo (a.k.a. sTypo or OS/2.typo) and win (or usWin) metrics. Depending on which OS you’re on and which application you’re in, a different set is used for rendering the font on the screen.

Unfortunately, all of these values relate to each other in a pretty complicated way. Fortunately, Glyphs does its best to calculate them based on the vertical metrics you enter for each of your masters: horní dotah, výška verzálek, střední výška and dolní dotah. You may run into problems, however, if these values change between masters.

Vlastní parametry

So, aby bylo možné avoid vertical jumps when you switch between different fonts of your family, it is a good idea to synchronize all values across all masters. You will do this with vlastní parametrs in File > Font Info > Masters (Cmd-I):

Set the values in one master, then copy and paste the parameters into the Custom Parameters fields of all other masters, or select all masters současně and batch-edit. If you have a separate file for your italics, do not forget to copy and paste the values to the other file as well.

But what do these values mean? Let me give you a quick rundown.

hhea

The name hhea refers to the hhea OpenType table. ‘hhea’ is supposed to be an abbreviation for Horizontal Header Table, meaning the general font info (a.k.a. the ‘header’) for horizontal typography like Latin, Hebrew, Greek, Javanese, Arabic or Thai. Apple devices such as Macs, iPhones, iPads, etc., use these values for rendering. The hhea table knows three vertical metrics values. For convenience, I will list them here with the vlastní parametr names that Glyphs uses:

Attention

On recent Apple systems, the typo values will be preferred over the hhea values if that font both contains a STAT table and Use Typo Metrics is turned on. Ve výchozím nastavení, all variabilní písmos exported by Glyphs contain a STAT table, and since Glyphs 3, Use Typo Metrics is on ve výchozím nastavení for all exports. You can force (or suppress) the export of a STAT table with an Export STAT table parameter in File > Font Info > Exports.

  1. hheaAscender: the height of the horní dotahs in units
  2. hheaDescender: the depth of the dolní dotahs in units (negative value)
  3. hheaLineGap: the recommended whitespace between lines

The hhea values have been preferred by Mac apps, and throughout the Apple realm (iPhones, iPads and the like). This is slowly changing, and more and more apps on the Mac will default to using the typo values instead (see below).

Web browsers that use the hhea values will add half of hheaLineGap on top of hheaAscender, and half below hheaDescender for the minimum <div> around a line of type. More details further below.

OS/2 sTypo (typo)

These values are part of the OS/2 OpenType table, or in its full name, the OS/2 and Windows Metrics Table. Anybody remember the operating system by the same name? Type pros commemorate it every day, thanks to vertical metrics. Sometimes, they are also referred to as sTypo or simply typo values, though. By the way, the s stands for ‘signed’, so the values can be both negative and positive. I have seen designers refer to them as ‘OS/2 values’, but that is a little imprecise, because the win values are also part of the OS/2 table.

  1. typoAscender: the height of the horní dotahs in units
  2. typoDescender: the depth of the dolní dotahs in units (negative value)
  3. typoLineGap: the recommended whitespace between lines

To quote Yannis Haralambous (p.724), these values ‘are oddly similar to ascent, descent, and lineGap in the hhea table but that should not necessarily be so precise or so closely tied to the vagaries of the glyphs’ outlines. Windows supposedly uses these values to find the ideal parameters for layout; thus we have a certain degree of artistic freedom.’ The general practice, however, is to keep typo and hhea values in sync, since different apps will use one of the two for exactly the same purposes: finding the first účaří, distance to subsequent účařís, and reserving rendering space above and below.

The big layout apps, XPress and InDesign, and other professional design software, can use the typoAscender value to determine the offset of the first účaří in a text box, and typoDescender for the minimum size of a text box below which the display of type is suppressed. However, most of these apps offer alternatives for calculating these offsets. In DTP apps, the line gap is meant to be controlled by the user, the default typically being 20% of the font size, hence typoLineGap is ignored.

Office software and browsers should prefer the typo values over the win values (see below) if the Use Typo Metrics parameter is set to yes, which it should always be in modern fonts. In Office apps, typoLineGap will be respected.

Again, web browsers that use the OS/2 typo values will add half of typoLineGap on top of typoAscender, and half below typoDescender for the minimum <div> around a line of type. But as always, there is more to it. More details further below.

The UPM dogma

There is one thing you need to watch out for, though: the ‘UPM dogma’. In the dark past of electronic typesetting, the TrueType/OpenType specifications used to stipulate that the span from typoAscender to typoDescender should be as large as the font’s UPM (usually 1000 or 2048). This used to make things really complicated.

A few years ago, the ‘UPM dogma’ has gotten under fire, because it simply did not work for scripts that require different metrics than Latin. One of the proponents of letting go of the UPM dogma was Victor Gaultney from SIL, who wrote both Best Practice: Design Metrics and Best Practice: Line Metrics.

In the meantime, the dogma was dropped altogether from the OpenType specification:

It is not a general requirement that sTypoAscender − sTypoDescender be equal to unitsPerEm. These values should be set to provide default line spacing appropriate for the primary languages the font is designed to support.

Yet, the UPM dogma still plays a role in the (legacy) Adobe and Microsoft strategies discussed below. But in modern fonts, you can safely ignore it.

OS/2 usWin (win)

The usWin, or in short, win values are also part of the OS/2 table. The us stands for ‘unsigned’, meaning that only positive values are possible.

  1. winAscent: the top extremum of the font rendering box
  2. winDescent: the bottom extremum of the font rendering box (positive value)

Attention: whatever extends above or below these values, will likely be cut off when rendered by the Windows text engine. Hence, the easiest way, it seems, is to ujistěte se everything in the font fits into the win span. So, usually, nothing you do in your font will exceed the win span.

Pro Tip

Aby bylo možné find the highest and lowest points in your font, try the mekkablue script Test > Report Highest and Lowest Glyphs. For every master, highest and deepest points in all glyphs will be listed.

The win values can be detached from the hhea and typo values. Firstly because they do different things: win takes care of clipping, whereas the others are supposed to be used for finding the účaří offsets, and by proxy, surrounding box dimensions. Secondly, the win values lack a line gap. However, if you do want to sync the win with the other value sets, you will have to set the hheaLineGap and typoLineGap values to zero.

Though, it is not so simple. In ancient times, Windows used the win values for everything and didn’t care much about typo values. See the next chapter.

If your font has support for a script that makes use of mark-to-base and/or mark-to-mark positioning, like Arabic vowel marks or the vowels in many Brahmic scripts, it is a good idea to also include extra space in the win values for potential mark placement. Otherwise you risk clipping of combining marks.

Use Typo Metrics

There is one more thing. If you can safely ignore older (i.e., pre-2006) MS Office versions, you should add a Use Typo Metrics parameter to File > Font Info > Font:

But if you forget, do not worry too much, because since Glyphs passed version number 3, Use Typo Metrics is on ve výchozím nastavení. In any event, if it is on, applications that respect this setting (in particular, versions of Microsoft Office since 2006) will prefer typoHorní dotah, typoDolní dotah, and typoLineGap over winAscent and winDescent for determining the vertical positioning.

Technically, what it does is set bit 7 (‘don’t use Win line metrics’) of the fsSelection field in the OS/2 table. According to the MakeOTF User Guide, this bit was introduced ‘so that reflow of documents will happen less often than if Microsoft just changed the behaviour for all fonts.’ Jinými slovy, all modern fonts should use typo metrics unless you have a very good excuse.

Alas, one good excuse may be that, with this parameter, legacy Office software (i.e., pre-2006) may apply clipping at the typo values rather than at the win values. If you find yourself in the position of needing to support ancient software: Duh. Everyone else is entitled to consider this a problem of the past.

Another problem is that, in Microsoft Office software, values for underlinePosition and underlineThickness are ignored when Use Typo Metrics is on. Even if it is off, and if the underline does not fit above winDescent, it will be raised accordingly. Jinými slovy, underlinePosition plus underlineThickness must be smaller than winDescent. If Use Typo Metrics is on, default underline values will be used instead. If you find the correct display of underlines in Microsoft Word more important than proper linespacing, disable Use Typo Metrics, and follow one of the legacy strategies. Most likely the Microsoft Strategy (see below) will be a good idea then.

Strategies

With the vlastní parametrs listed above, you can override the automatic calculation and set the values manually. Let me show you the most popular strategies for manually setting your vertical metrics. I am also listing two historical methods, the Adobe and Microsoft strategies. They are handy to know because you may have to make your font compatible with legacy software. But mějte na paměti that both of these strategies are outdated because they both adhere to the UPM dogma. Therefore, if you need a good overall compromise for DTP, Office and web use, I recommend to employ the Webfont strategy described below.

Webfont strategy (2019)

Here, you set the winAscent and winDescent values first, because what is clipped and what not is most important on Windows machines.

Tip

We support and recommend the webfont strategy. In our opinion, it is the most compatible compromise for vertical metrics.

On the Mac, the Safari and Chrome browsers use the hhea values for positioning text, regardless of the Use Typo Metrics setting. When text is placed inside an HTML element such as <div> or <p>, these browsers will add hheaAscender plus half of hheaLineGap, and use this to calculate the position of the first účaří in respect to the top edge of the HTML element. Similarly, the distance from the very last line of text inside an HTML element to the bottom edge of the same element is determined by hheaDescender plus half of hheaLineGap. That’s right, the line gap is distributed evenly above and below the line.

Notable exception on the Mac: Firefox respects the Use Typo Metrics setting, and will do the same with the OS/2 typo metrics if it is set, i.e., the first účaří will be put at typoAscender plus half typoLineGap, and the space below the last účaří is equal to typoDescender plus half typoLineGap. If, however, Use Typo Metrics is not set, it will act like the other browsers on the Mac and employ the hhea values.

On Windows, all browsers respect the Use Typo Metrics parameter. If it is set, first účaří will be positioned at typoAscender plus half typoLineGap, and the distance between last účaří and bottom edge will be typoDescender plus half typoLineGap. If, however, Use Typo Metrics is not set, then all Windows browsers will default to the win values. V takovém případě, the first účaří will be at winAscent from the top edge, and the bottom edge will be padded winDescent away from the last účaří.

As a consequence, if we do make use of the Use Typo Metrics parameter as we are supposed to, the win values are now completely independent of the hhea and typo values. So we can use the hhea and typo values for what they were intended for, including the line gap. Simply set the win values to the vertical extremes in your font family, and ujistěte se, like in the Adobe strategy, to sync typo and hhea values. So what we get is this:

  1. winAscent = vertical maximum round this value up
  2. winDescent = vertical minimum (positive value) round this value up
  3. typoAscender = hheaAscender = include important uppercase diacritics (e.g. É, Ñ, Ő, etc.) or the letters reaching high above the účaří you care most about, it is okay to cut a bit into the highest diacritics (typically Å in Latin or a Vietnamese double diacritic) round this value
  4. typoDescender = hheaDescender = include dolní dotahs completely (the lowest point in j, g, p, q, y, or the letters reaching below the účaří you care most about) round this value down
  5. typoLineGap = hheaLineGap = sensible padding between lines: approx 10–20% of the sum of typoAscender and typoDescender, consider more if dolní dotahs and accents touch each other across lines (in browsers and Office software), less if your horní dotah and dolní dotah values are pretty large
  6. Font Info > Font > Custom Parameters: Use Typo Metrics = yes

About finding a proper line gap value (point 5): Simply imagine a word on a button or in a box on a web page. Half of the line gap amount will be padded above, and half below the (typo and hhea) horní dotah and dolní dotah positions.

And if, for whatever reason, you cannot or do not want to enable Use Typo Metrics, you can try:

typoLineGap = hheaLineGap = (winAscent − typoAscender) × 2

The offset of the first účaří will be consistent even without the Use Typo Metrics parameter. That may make sense if you want to support legacy software as described above. However, the leading may still differ unless the difference between winDescent and typoDescender happens to be exactly the same as the difference between winAscent and typoAscender.

If the calculations lead to large line gap values (anything larger than a fifth of the UPM), consider reducing the line gaps and increasing the hhea and typo horní dotahs by the same value.

The Google strategy

Most recently, Google has been proposing a new set of guidelines for vertical metrics. They aim for the maximum consistency possible across all platforms, apps and environments. Because not all apps respect them, line gaps have to be set to zero.

  1. hheaLineGap = typoLineGap = 0
  2. hheaAscender = typoAscender = height of highest cap diacritic in family, typically Abreveacute
  3. hheaDescender = typoDescender = negative value, depth of deepest letter, typically one of g, j, p, q, or y
  4. Adjust the horní dotah or, more likely, the dolní dotah values in such a way that all-cap type set in a box defined by these values appears vertically centered in the box, so that typoAscender − capHeight = −typoDescender
  5. The span from horní dotah to dolní dotah should be at least 20% greater than UPM, increase both if necessary
  6. winAscent = vertical maximum plus some extra space for accommodating combining top marks if that makes sense for your font
  7. winDescent = positive value, vertical minimum and perhaps extra space for combining bottom marks if that makes sense for your font
  8. Font Info > Font > Custom Parameters: Use Typo Metrics = yes

Even though the Google strategy prefers caps, I would say: do find a good compromise for both all-cap and mixed-case setting when centering. Blindly following the calculation will make mixed-case type look a bit too low.

Legacy strategies

Every once in a while, you will come across old fonts that do not have the Use Typo Metrics bit set (because it had not come into existance yet), and where the display on Mac, Windows and DTP apps will only partially be in sync. That is because these values use to serve different purposes in the dark past of pre-OpenType digital typography. Plus, the UPM dogma for the typo values was still very much alive and kicking. So for historical reasons, and for the rare case that you will need to work on a legacy font, or make a font for a legacy environment, I will list the two main strategies, reverse-engineered from the cold dead hands of dusty old TTFs.

The Adobe strategy

The hhea values are synchronized with the typo values.

  1. hheaAscender = typoAscender
  2. hheaDescender = typoDescender
  3. hheaLineGap = typoLineGap = winAscent + winDescent – UPM
  4. typoAscender − typoDescender = UPM (‘UPM dogma’)
  5. Use Typo Metrics = off

With this strategy, the line gap tends towards the small end of the spectrum, also closer to how PS Type 1 fonts used to store their vertical metrics, and more suitable for the needs of layout apps. So it may be a good idea to use this method if your font has a low střední výška (below half UPM), or more generally put, it may be good for designs where the distance between lowest and highest node is about the UPM. Advantages: better synchronization of font display between Mac apps and layout apps (XPress, InDesign), and usually tight default leading. Disadvantages: vertically pretty tight due to the UPM dogma, accents on caps may be cut off in office apps, difference between Mac and Windows because Windows would use the win values for vertical layout.

The Microsoft strategy

The hhea values are synchronized with win values, thus to the maxima of the font bounding box (the smallest box around all glyphs of the font superimposed).

  1. hheaAscender = winAscent
  2. hheaDescender = −winDescent
  3. hheaLineGap = 0
  4. typoLineGap = winAscent + winDescent – UPM
  5. typoAscender − typoDescender = UPM (‘UPM dogma’)
  6. Use Typo Metrics = off

For the rest, as already mentioned above, the span from typoAscender to typoDescender must add up to your UPM value (usually 1000). You could put the depth of the dolní dotah into typoDescender (e.g. −200), you put the rest (800) into typoAscender. In this strategy, the sum of the hhea and win horní dotah and dolní dotah will most likely be much more than 1000, or whatever your UPM is. Subtract your UPM value from that sum and put the result into typoLineGap.

With this strategy, the linegap tends towards the large end of the spectrum. So it may be a good idea to use this way if your font has a large střední výška (above half UPM). Advantages: better synchronization of font display between Win and Mac apps; accents are not cut off in Mac apps because winAscent tends to be higher than typoAscender. Disadvantage: differences between Mac apps and layout apps (XPress, InDesign), and default leading may appear to be too much.

First účaří in Adobe apps

You may do everything right, and still get complaints from users, particularly about the positioning of the first line of text in a text frame. In InDesign and Illustrator, the the first účaří offset depends on the settings in the respective document.

The weirdest thing though is that the default setting, ‘Ascent’, is the measurement of the Latin lowercase d. So, if you need to ujistěte se that your font aligns well with others, and you do not want to spend a lot of your precious lifetime on explaining to Adobe users the two dialogs below, consider synchronising the heights of your lowercase d’s.

Especially if you are making a layered font, and the shapes must align, you may need to make use of the lowercase-d hack. See the Layered Color Font tutorial for details.

InDesign:

In InDesign, select a text frame, and choose Object > Text Frame Options… (Cmd-B), and in the dialog, pick the Účaří Options tab at the top, and where it says First Účaří, you have the following Offset options:

  • Ascent: The height of the “d” character in the font falls below the top inset of the text frame.
  • Cap Height: The top of uppercase letters touch the top inset of the text frame.
  • Leading: Use the text’s leading value as the distance between the účaří of the first line of text and the top inset of the frame.
  • x Height: The height of the “x” character in the font falls below the top inset of the frame.
  • Fixed: Specify the distance between the účaří of the first line of text and the top inset of the frame.
  • Min: Select a minimum value for the účaří offset. Například, if Leading is selected and you specify a minimum value of 1p, InDesign uses the leading value only when it’s greater than 1 pica.

Find more info about InDesign text frames on the Adobe help page.

Illustrator:

In Adobe Illustrator, select a text frame and choose Type > Area Type Options…, and in the dialog that pops up, pick an option in Offset > First Účaří:

  • Ascent: The height of the “d” character falls below the top of the type object.
  • Cap Height: The tops of uppercase letters touch the top of the type object.
  • Leading: Uses the text’s leading value as the distance between the účaří of the first line of text and the top of the type object.
  • x Height: The height of the “x” character falls below the top of the type object.
  • Em Box Height: The top of the em box in Asian fonts touches the top of the type object. This option is available regardless of the Show Asian Options preference.
  • Fixed: Specifies the distance between the účaří of the first line of text and the top of the type object in the Min box.
  • Legacy: Uses the first účaří default used in Adobe Illustrator 10 or earlier.

Find more info about Illustrator text frames on the Adobe help page.

TT Zones

If you are exporting TTFs and still experience cut-offs in apps like Microsoft Word, especially with double accents (like the ones in Vietnamese or Pinyin), then try this: First ujistěte se your winAscent and winDescent are properly set, e.g., they encompass the highest and lowest glyphs you want to keep from being cut off. And now, you need TT zones at winAscent and winDescent. These additional zones cause the renderer to include everything up to their position.

If you are manually TrueType hinting, you can add the winAscent and winDescent zones with the TTF Zones parameter in File > Font Info > Masters (Cmd-I).

If, however, you are relying on ttfautohint, there is an even easier way. All you need to do is go to File > Font Info > Exports > Custom Parameters enable the Windows Compatibility option in the TTFAutohint options parameter. Repeat that for every TTF instance, and you are done:

Useful plug-ins and scripts

The plug-in View > Show Vertical Metrics can visualize your vlastní parametrs in Edit view. You can find it in Window > Správce zásuvných modulů:

In the mekkablue scripts, you will find Font Info > Vertical Metrics Manager and Test > Report Highest and Lowest Glyphs.

The Vertical Metrics Manager will try and apply the Webfont Strategy as much as possible. It can measure any set of glyphs and determine viable values to some extent. You can edit the values yourself, before you apply them to the font. Extensive documentation is available in the tooltips: if there is a thing you do not understand, just hover the mouse over it for a second or two. Once you run the script functions, the script will document its process in Window > Macro Window.

Report Highest and Lowest Glyphs simply spits out the most extreme glyphs in terms of vertical extension, and writes a small report in the Macro Window. This can be useful for finding good win values.
